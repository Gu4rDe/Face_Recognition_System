## Контекст

Приложение KotlinApp — JVM Compose Desktop-клиент для системы распознавания лиц. Текущий экран «Лица» (FaceRecognitionScreen) позволяет только выбрать фото из файла через JFileChooser, отправить на `POST /api/v1/faces/recognize` и отобразить результаты в виде текстовых карточек (имя, сходство в %, координаты bbox числами). В приложении уже существует инфраструктура захвата с камеры (WebcamService + PhotoCaptureDialog), используемая только при регистрации сотрудников. Настройки приложения (SettingsOverlay) содержат лишь тему и URL API, хотя модель AppSettings включает matchThreshold, cameraResolution, cameraFps.

Стейкхолдеры: операторы системы (нужны визуальная обратная связь и настройка порога), администраторы (настройка параметров камеры).

## Цели / Без целей

**Цели:**
- Обеспечить захват фото с веб-камеры на экране распознавания лиц без дублирования кода
- Визуализировать bounding box лиц поверх изображения предпросмотра с подписями совпадений
- Дать операторам возможность настраивать порог распознавания и параметры камеры из UI
- Переиспользовать существующие WebcamService и PhotoCaptureDialog без их модификации

**Без целей:**
- Реализация on-device ML-распознавания (всё остаётся серверным)
- Изменение API контракта (DTO, эндпоинты)
- Модификация экрана регистрации сотрудников
- Поддержка нескольких камер (используется Webcam.getDefault())

## Решения

### 1. Интеграция веб-камеры через PhotoCaptureDialog

**Решение**: Добавить кнопку «Снять с камеры» в FaceRecognitionScreen. При нажатии — показать PhotoCaptureDialog, получить ByteArray и использовать как источник для распознавания.

**Альтернатива**: Создать отдельный диалог захвата. Отклонено — дублирует код PhotoCaptureDialog с теми же функциями (стриминг, захват, политика согласия).

**Обоснование**: PhotoCaptureDialog уже реализует полный цикл (открытие камеры → стриминг → захват → согласие → возврат ByteArray). Единственное различие — caller, а не логика диалога.

### 2. Отрисовка bounding box через Canvas-overlay composable

**Решение**: Создать `FaceBoundingBoxOverlay` — Composable-функцию поверх Image, использующую `Canvas` для отрисовки прямоугольников и текстовых подписей.

Детали реализации:
- Использовать `Modifier.drawWithContent` / `Canvas` поверх `ImageBitmap`
- Масштабировать координаты bbox из оригинального размера изображения в отображаемый размер (через `ContentScale.Fit` — вычисление коэффициентов)
- Цветовые коды: сходство >80% — зелёный, 60–80% — жёлтый/оранжевый, <60% — красный
- Подпись: «{username} ({similarity%})» над прямоугольником
- Если совпадений нет — показывать только прямоугольник без подписи

**Альтернатива**: Использовать Overlay-layout с Box'ами. Отклонено — рисование на Canvas точнее и производительнее, не требует вычисления layout-позиций.

### 3. Настройки распознавания в SettingsOverlay

**Решение**: Расширить `SettingsState` наблюдаемыми полями (`matchThreshold`, `cameraResolution`, `cameraFps`), загружать их при открытии диалога через `SettingsRepository.getSettings()`, сохранять через `SettingsRepository.updateSettings()`.

UI-компоненты:
- `matchThreshold`: Slider (0.0–1.0, шаг 0.01) + текстовое отображение текущего значения в %
- `cameraResolution`: OutlinedTextField (строка, напр. «640x480»)
- `cameraFps`: OutlinedTextField (целое число)

**Альтернатива**: Хранить настройки только локально. Отклонено — текущая архитектура хранит настройки на сервере, и порог matchThreshold используется сервером при распознавании.

## Риски / Компромиссы

- **[Риск] Масштабирование bbox при разных размерах Image**: Координаты bbox приходят в пикселях оригинального изображения. При `ContentScale.Fit` необходимо точно вычислять смещения (letterboxing). → Митигация: вычислять оригинальные размеры через `ImageBitmap.width/height`, применить формулу масштабирования ContentScale.Fit с учётом aspect ratio.

- **[Риск] Производительность отрисовки overlay при большом числе лиц**: Canvas перерисовывается при каждом recomposition. → Митигация: Bounding box отрисовывается только после получения результата, не при каждом кадре. Число лиц обычно ≤ 10 — не критично.

- **[Риск] WebcamService не закрывается при навигации**: Если пользователь переключит экран во время открытого PhotoCaptureDialog, камера может остаться занятой. → Митигация: PhotoCaptureDialog уже вызывает `WebcamService.close()` в `DisposableEffect.onDispose`. Это покрывает случай навигации.

- **[Компромисс] Настройки применяются при закрытии диалога**: Как и текущий паттерн с API URL — изменения сохраняются по кнопке «Закрыть» или при dismiss. Не apply-on-change, что упрощает логику.
