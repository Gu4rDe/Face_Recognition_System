## Context

Добавление сотрудника в текущей реализации использует JFileChooser для выбора фото и содержит лишние поля формы (employee_id, location, hire_date, isActive). Приложение работает на JVM/Desktop с Compose Multiplatform. Проект использует Kotlin, Compose Multiplatform, Voyager, Ktor для HTTP.

## Goals / Non-Goals

**Goals:**

- Убрать employee_id из формы, генерировать автоматически (EMP-{timestamp})
- Убрать поля location, hire_date и переключатель isActive из формы
- Заменить JFileChooser на диалог захвата с веб-камеры
- Обеспечить предпросмотр камеры в реальном времени в диалоге
- Добавить обязательный чекбокс политики использования с кликабельной ссылкой
- Блокировать кнопку Сохранить до принятия политики

**Non-Goals:**

- Запись видео — только одиночный кадр (фото)
- Хранение текста политики на сервере — текст статичный, зашит в приложение
- Повторное использование диалога для других целей (пока только фото сотрудника)
- Мобильная платформа — webcam-capture работает только на JVM

## Decisions

### D1: Библиотека webcam-capture (sarxos)

**Решение:** Использовать com.github.sarxos:webcam-capture для доступа к веб-камере.

**Альтернатива:** JavaCV/OpenCV — слишком тяжёлая зависимость для задачи захвата одного кадра.

**Обоснование:** webcam-capture — лёгкая библиотека (нет транзитивных зависимостей кромеbridg), простое API, стабильна, активно поддерживается, работает на JVM/desktop.

### D2: WebcamService как singleton

**Решение:** Создать WebcamService (object) с методами open(), capture(), close(). Сервис управляет жизненным циклом веб-камеры: открытие при показе диалога, захват по нажатию кнопки, закрытие при закрытии диалога.

**Обоснование:** Изолирует работу с камерой от UI, предотвращает утечку ресурсов, позволяет переиспользовать в будущем.

### D3: PhotoCaptureDialog как отдельный composable

**Решение:** PhotoCaptureDialog — отдельный Composable-компонент в пакете ui/dialogs/. Содержит: предпросмотр камеры в реальном времени, кнопку Захватить, предпросмотр захваченного фото, чекбокс политики, кнопки Сохранить/Отмена.

**Обоснование:** Вынос в отдельный компонент упрощает тестирование и переиспользование. Dialog из Material3 обеспечивает модальность.

### D4: Автогенерация employee_id

**Решение:** employee_id генерируется как `EMP-{System.currentTimeMillis()}` в момент вызова createEmployee. Форма не содержит поле для ввода.

**Обоснование:** Идентификатор уникален (включает миллисекунды), формат согласуется с тестами (ServerCommunicationTest использует EMP-{timestamp}).

### D5: Политика использования — кликабельная ссылка в чекбоксе

**Решение:** Чекбокс с текстом `Я принимаю [Политику использования]`, где `Политику использования` — кликабельный текст (HyperlinkText), открывающий URL в браузере или показывающий AlertDialog с текстом политики.

**Обоснование:** Полный текст политики слишком длинный для диалога захвата. Ссылка позволяет ознакомиться без перегрузки UI.

### D6: Закрытие камеры в onDispose

**Решение:** При закрытии PhotoCaptureDialog (onDispose) вызывается WebcamService.close() для освобождения ресурсов камеры.

**Обоснование:** Предотвращает утечку ресурсов. Камера не должна оставаться открытой после закрытия диалога.

## Risks / Trade-offs

- **[webcam-capture зависимости]** Добавляется новая зависимость 280KB. -> Приемлемо для desktop-приложения.
- **[Камера не найдена]** На устройствах без камеры диалог должен показывать сообщение об ошибке. -> WebcamService возвращает null при отсутствии камеры.
- **[Поток камеры в Compose]** Вывод кадров камеры в Compose требует конвертации BufferedImage -> ImageBitmap. -> Использовать SkiaImage.makeFromEncoded() как в текущей реализации предпросмотра.
