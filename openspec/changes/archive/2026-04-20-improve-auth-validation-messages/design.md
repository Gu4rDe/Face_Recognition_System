## Context

Экраны LoginAdminScreen и RegisterAdminScreen показывают минимальные ошибки валидации ("Введите логин"). ErrorMapper переводит HTTP-коды в общие сообщения без инструкций. Нет подсказок под полями.

Текущие файлы:
- `LoginAdminScreen.kt` — валидация только на `isBlank()`
- `RegisterAdminScreen.kt` — валидация только на `isBlank()`
- `ErrorMapper.kt` — маппинг HTTP-кодов без helpful-инструкций

## Goals / Non-Goals

**Goals:**
- Показывать конкретные ошибки валидации для каждого поля (длина, формат, символы)
- Добавить подсказки (hint) под полями с требованиями
- Добавить инструкции после серверных ошибок (что делать)
- Вынести логику валидации в переиспользуемые функции

**Non-Goals:**
- Изменять API сервера
- Добавлять новые экраны
- Реализовывать восстановление пароля

## Decisions

**Decision 1: Создать утилитарный объект FormValidator**

Альтернативы:
1. Inline-валидация в каждом экране — дублирование, сложно тестировать
2. Отдельный объект `FormValidator` с функциями-валидаторами — чисто, переиспользуемо, тестируемо

Выбираем вариант 2. Объект в пакете `util` с функциями: `validateUsername()`, `validateEmail()`, `validatePassword()`, `validateInviteCode()`. Каждая возвращает `String?` (null = валидно, строка = сообщение об ошибке).

**Decision 2: Подсказки через SupportingText в OutlinedTextField**

Использовать параметр `supportingText` у `OutlinedTextField` для показа hint-текста серым цветом, а ошибку — красным. Это стандартный паттерн Material 3.

**Decision 3: Инструкции после серверных ошибок через ErrorMapper**

Обновить ErrorMapper чтобы он возвращал ошибку + инструкцию на новой строке для ключевых кодов:
- 401 → "Неверный логин или пароль.\nПроверьте правильность ввода и попробуйте снова"
- 403 → detail + "\nУбедитесь, что ваш код приглашения действителен"
- 409 → "Пользователь уже существует.\nПопробуйте другой логин или email"

## Risks / Trade-offs

[Дублирование hint-текста и ошибок] → Формулы hint подсказок единообразны, ошибки в FormValidator — митигация через централизацию
[Расхождение с серверной валидацией] → Клиентские правила могут быть мягче серверных — показываем detail из ответа сервера как fallback
